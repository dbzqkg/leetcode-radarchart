# 第一部分：系统架构设计书 (Architecture Specification)

本架构设计书仅定义本原型系统的抽象模块、调用关系、数据流向模型与生命周期，**不包含任何具体第三方类库、API 路径及代码级细节**。

## 1. 系统边界与非目标 (Scope & Boundaries)

### 1.1 系统目标

接收客户端提供的验证凭证，通过中转获取第三方目标平台的分类解答指标，并将计算进度与最终结果流式反馈至前端画布进行呈现。

### 1.2 绝对执行的非目标 (Strict Non-Goals)

* **零状态存储**：不提供任何形式的持久化与缓存机制。
* **单并发串行同步**：不进行高并发线程池维护，每次处理请求均采用极简的单任务同步步进模式。
* **无防御性代码**：不设计请求重试机制、不设延迟抖动，遇到任何网络或凭证异常一律立即强行中断连接。
* **无用户或账户关联**：不保存、不解析、不跟踪任何用户登录态或归属关系。

## 2. 核心模块与调用关系 (Component Relationship)

本系统由四个抽象单元组成：

```text
 [前端展示单元] (Visual Component)
       │▲
 (单向流式通信)
       ▼│
 [中转控制器] (Bridge Controller) ── 承接会话并脱离主线程
       │
 (串行分步调用)
       ▼
 [数据提取器] (Data Fetcher) ── 代理发送网络调用
       │
 (结构化提取)
       ▼
 [数据转换器] (Data Converter) ── 解析原始结构为扁平数据
```

* **前端展示单元 (Visual Component)**：负责捕获用户凭证、提供进度反馈渲染、承载雷达图画布。
* **中转控制器 (Bridge Controller)**：作为前后端长连接的端点，负责在收到请求后脱离主线程，启动单向异步分发流。
* **数据提取器 (Data Fetcher)**：负责伪装网络调用，代理前端向第三方数据源发起串行数据抓取。
* **数据转换器 (Data Converter)**：负责提取原始响应体中目标分类的解答总数，向控制器输出强契约的扁平数值。

## 3. 抽象数据流模型 (Abstract Data Flow)

1. **连接握手**：客户端发起承载凭证的连接，后端控制器承接并立即向客户端返回单向流通道。
2. **串行抓取**：工作线程按分页方式遍历用户所有已解题目，每页最多 100 题，向数据提取器发出分页指令。
3. **提取与转换**：数据提取器返回原始报文，由数据转换器提取每道题目的 topicTags 标签列表，按标签名称聚合统计各标签的解题数量（含简单/中等/困难分布），并返回给控制器。
4. **状态反馈**：控制器将步骤进度实时向流通道推送。
5. **会话终结**：最后一个分类数值收集并推送成功后，后端主动声明流通道结束并销毁所有会话堆栈。

## 4. 临时生命周期模型 (Session Lifecycle)

整个计算会话的生命周期与底层的 HTTP 长连接生命周期完全绑定。一旦连接由于数据传输完毕、超时或遇到任何异常而关闭，临时堆栈及凭证变量的生命周期即宣告结束，内存空间等待即时释放。

# 第二部分：技术选型书 (Technology Selection Specification)

本技术选型书仅回答“用什么技术解决什么问题，以及为什么排斥替代方案”，**不包含任何数据报文结构、方法签名、逻辑循环或特定接口细节**。

## 1. 后端技术选型决策 (Backend Stack Decision)

### 1.1 开发平台：Spring Boot 3.x + Java 21

* **解决问题**：快速构建可对外提供 Web 端点的 HTTP 服务，并依靠 Java 强类型约束向生成代码的 Agent 提供明确的数据类型契约。
* **替代排查**：为什么不用 Node.js？Node.js 虽然轻量，但其动态类型特性在 Agent 自动生成代码时容易导致数据契约混淆和幻觉报错。Java 21 的强类型和 Record 记录类能保证 Agent 生成的代码结构 100% 稳定。

### 1.2 网络调用代理：JDK Native HttpClient

* **解决问题**：充当服务端对服务端的网络代理（中转），用以直接绕过浏览器的同源策略（CORS 跨域限制）。
* **替代排查**：为什么不用 Apache HttpClient 或 OkHttp？因为 JDK 原生自带的 `java.net.http.HttpClient` 足够轻量，无需引入任何额外的 Maven 外部依赖，降低了 Agent 处理外部包冲突和版本兼容问题的几率。

### 1.3 进度流推送：Spring SseEmitter (Server-Sent Events)

* **解决问题**：实现后端向前端的低开销、单向、渐进式进度推送。
* **替代排查**：为什么不用 WebSocket？WebSocket 拥有全双工能力，但协议握手繁琐，在本原型“后端仅单向汇报进度”的极简场景下，技术复杂度过剩。

### 1.4 JSON 解析：Jackson Tree Model

* **解决问题**：非侵入式解析力扣 GraphQL 的复杂返回体。
* **替代排查**：为什么不建立完整的 POJO 实体映射？因为本原型不进行持久化，Jackson 的 Tree Model（`JsonNode`）可以直接定位特定路径下的叶子节点，保持代码扁平化，避免引入多余的冗余 DTO。

## 2. 前端技术选型决策 (Frontend Stack Decision)

### 2.1 核心框架：Vue 3 (Composition API) + Vite

* **解决问题**：快速提供用户交互表单和进度展示画布。
* **替代排查**：为什么不用 React？在快速生成 MVP 工具页面时，Vue 3 的单文件组件（SFC）把 HTML/JS/CSS 逻辑高度聚合，非常有利于 AI 编程助手进行单文件全套代码的直接输出，避免跨组件拼装错误。

### 2.2 图表呈现：Apache ECharts (Radar Chart)

* **解决问题**：渲染直观的五维能力雷达图，并支持自适应展示。
* **替代排查**：为什么不用 Chart.js 或 D3？ECharts 拥有最适合雷达图展示的原生配置集，样式更美观，且可以通过纯配置项声明，降低了 Agent 编写图表逻辑的代码复杂度。

### 2.3 数据流消费：Fetch API (ReadableStream Reader)

* **解决问题**：安全地通过 POST 方法将用户验证凭证作为载荷发送，并实时解包后端的流式数据。
* **替代排查**：为什么不用原生的 EventSource？因为原生 EventSource 协议强制限制必须使用 GET 方法，这会导致敏感的 Cookie 凭证不得不拼接到 URL 的 Query 参数中，带来网络链路泄露的安全风险。

# 第三部分：细节实现书 (Implementation Detail Specification)

本细节书专为 AI Agent 代码生成编写，包含具体接口路径、JSON schema、具体方法处理流程与伪代码，**不包含任何架构设计解释或技术选型讨论**。

## 1. API 接口与网络契约定义 (API Contract)

### 1.1 后端暴露的连接端点

* **路径**：`POST /api/analyze/sse`
* **Content-Type**：`application/json`
* **请求体 JSON Schema**：

    ```json
    {
      "type": "object",
      "properties": {
        "cookie": { "type": "string" }
      },
      "required": ["cookie"]
    }
    ```

* **响应头**：`Content-Type: text/event-stream`

### 1.2 力扣第三方数据源端点与载荷

* **目标 URL**：`https://leetcode.cn/graphql/`
* **网络方法**：`POST`
* **GraphQL 请求 JSON Payload**：

    ```json
    {
      "operationName": "problemsetQuestionList",
      "variables": {
        "categorySlug": "",
        "skip": 0,
        "limit": 100,
        "filters": { "status": "AC" }
      },
      "query": "query problemsetQuestionList($categorySlug: String, $limit: Int, $skip: Int, $filters: QuestionListFilterInput) { problemsetQuestionList(categorySlug: $categorySlug, limit: $limit, skip: $skip, filters: $filters) { hasMore total questions { difficulty topicTags { name nameTranslated } } } }"
    }
    ```

* **JSON 提取目标说明**：
    * `$.data.problemsetQuestionList.total`：获取已解题目总数（整数）。
    * `$.data.problemsetQuestionList.hasMore`：判断是否有下一页（布尔值）。
    * `$.data.problemsetQuestionList.questions[*].difficulty`：题目难度（EASY / MEDIUM / HARD）。
    * `$.data.problemsetQuestionList.questions[*].topicTags[*]`：该题关联的算法标签数组，每项含 `nameTranslated`（中文名，优先）和 `name`（英文名） [1]。

## 2. 核心处理流程伪代码 (Core Processing Logic)

### 2.1 控制器接收与验证流程 (`SseEmitterController`)

1. 解析前端 POST 数据包中的 `cookie` 字段。
2. 验证 Cookie 字符串中是否包含 `"LEETCODE_SESSION"` 子串。
    * 若不包含，立即抛出 `IllegalArgumentException`，终止方法，返回 HTTP 400 [1]。
3. 初始化 `SseEmitter emitter = new SseEmitter(180000L)` [1]。
4. 异步调度抓取序列（将 `cookie` 与 `emitter` 传入抓取服务）。
5. 立即返回 `emitter` 实例给前端。

### 2.2 串行代理抓取流程 (`LeetCodeCrawlerService`)

1. 初始化分页变量：`int skip = 0;` `int limit = 100;` `int loaded = 0;` `int totalQuestions = 0;`
2. 声明标签统计容器：`Map<String, TagStat> tagMap = new LinkedHashMap<>();`（TagStat 含 name / total / easy / medium / hard 字段）。
3. 实例化局部调用客户端：`HttpClient client = HttpClient.newHttpClient();`
4. **串行分页循环**（`while (hasMore)`）：
    * **4.1 构造 GraphQL 请求体**：`categorySlug` 固定为空字符串（抓取全站），`filters` 固定 `{ "status": "AC" }`，`skip` 和 `limit` 由分页变量决定。
    * **4.2 装配并发出请求**：
        * 添加 Header `Cookie`: 传入的 `cookie` 字符串。
        * 添加 Header `User-Agent`: `Mozilla/5.0 (Windows NT 10.0; Win64; x64)...`
        * 添加 Header `Referer`: `https://leetcode.cn/`
        * 添加 Header `Origin`: `https://leetcode.cn`
        * 发送 POST 请求至 `https://leetcode.cn/graphql/`
    * **4.3 解析响应**：
        * 若 HTTP 状态码不等于 200，或者 JSON 返回体包含 `"errors"` 键：
            * 构建异常消息：`{"status": "ERROR", "message": "凭证失效或请求受限"}`。
            * 向 `emitter` 推送该消息，调用 `emitter.complete()`，直接退出线程。
        * 若正常响应，提取 `total`（总题数）、`hasMore`（是否有下一页）。
    * **4.4 遍历本页题目进行标签聚合**：
        * 对每道题解析 `difficulty` 字段。
        * 遍历 `topicTags` 数组，提取 `nameTranslated`（优先）或 `name` 作为标签名。
        * 若标签名不存在于 `tagMap`，初始化 TagStat 实体。
        * 累加该标签的 total 及对应难度（EASY/MEDIUM/HARD）计数。
        * 加载计数器 `loaded++`。
    * **4.5 反馈实时进度**：
        * 向 `emitter` 推送流消息：`{"status": "RUNNING", "message": "正在加载... (loaded/totalQuestions)", "currentStep": loaded, "totalSteps": totalQuestions}` [1]。
    * **4.6 更新分页并同步停顿**：
        * `skip += LIMIT;`
        * 执行线程硬睡眠：`Thread.sleep(500L);` [1]。
5. **排序并推送最终数据**：
    * 将 `tagMap` 的值按 `total` 降序排列。
    * 组装 SUCCESS 报文体：`{"status": "SUCCESS", "data": {"total": totalQuestions, "tags": [{"name": "...", "total": N, "easy": N, "medium": N, "hard": N}, ...]}}` [1]。
    * 向 `emitter` 发送该报文，随即调用 `emitter.complete()` 关闭单向流通道 [1]。

### 2.3 前端流消费与雷达绘制逻辑 (`App.vue`)

1. **流读取循环**：
    * 前端使用 `fetch` 建立连接，解码并读取字节流：

        ```javascript
        const reader = response.body.getReader();
        const decoder = new TextDecoder();
        while (true) {
          const { value, done } = await reader.read();
          if (done) break;
          // 按 \n 字符切分流片段，并解析带有 "data:" 前缀的 JSON 字符串
          // ... 提取 payload 对象
          if (payload.status === "RUNNING") {
             // 绑定并更新界面组件的进度数值
             progressPercent.value = (payload.currentStep / payload.totalSteps) * 100;
             statusText.value = payload.message;
          } else if (payload.status === "SUCCESS") {
             // 流结束，更新雷达图配置
             drawRadar(payload.data);
             break;
          }
        }
        ``` [1]
2. **雷达图动态自适应计算与绘制**：
    * 从 `payload.data.tags` 取前 10 个标签（按总数降序排列） [1]。
    * 计算当前最大值 `const rawMax = Math.max(...vals, 1)`。
    * 雷达图单轴最大值（`max`）算法设定：
        `const computedMax = rawMax < 10 ? 10 : Math.ceil((rawMax * 1.2) / 10) * 10;`
    * 渲染 ECharts 配置：`radar.indicator` 由标签名动态生成，`max` 统一设为 `computedMax`，`series[0].data` 填入对应数值 [1]。
    * 图表高度随标签数量自适应：`style.height = (200 + display.length * 40) + 'px'` [1]。
    * 下方渲染完整标签统计表格，展示所有标签的 total / easy / medium / hard 分布，超出前 10 的标签半透明显示 [1]。
